# Incident: 2026-04-07 13:58

## Summary

At 13:58 UTC on April 7, 2026, pizza order requests to `jwt-pizza-service` began returning HTTP 500 responses because the downstream pizza factory service was intentionally put into a chaos state. The failure originated in the factory order API, which responded with `{"message":"chaos monkey"}`, and `jwt-pizza-service` surfaced that as `{"message":"Failed to fulfill order at factory"}` to clients. This was a low-severity, test-induced incident used to validate monitoring, alerting, and incident response during a controlled chaos exercise.

The incident affected order placement only and lasted from the time chaos was enabled until it was disabled and error rates returned to baseline. Detection came from the on-call alert configured for elevated order failures / HTTP 500s. No evidence was provided of customer support tickets or external escalation during the test window.

## Detection

This incident was detected when the on-call alert for elevated order failures triggered after chaos was enabled in the pizza ordering path. Boston confirmed the incident by reviewing application logs showing a `POST /api/order` request returning status code `500` and a corresponding downstream `factory_request` returning status code `500`.

Time-to-detection could be improved by alerting directly on the ratio of failed order submissions to total order attempts, rather than relying only on general HTTP 500 volume. That would make the signal more specific to the business action that failed and reduce the time required to identify the affected dependency.

## Impact

For the duration of the chaos test beginning at 13:58 UTC on 2026-04-07, users attempting to place pizza orders experienced failed checkout attempts when the downstream pizza factory returned errors.

Affected users received an HTTP 500 response from `POST /api/order` with the message `Failed to fulfill order at factory`. Based on the chaos design, the impact was intermittent rather than total, depending on which requests hit the injected failure path. No support tickets, social media reports, or customer escalations were captured for this controlled exercise.

## Response

After the alert fired, Boston reviewed the order and factory logs and identified that the failure was not caused by malformed client input, but by the downstream pizza factory returning a `500` during order fulfillment.

Boston correlated the two log events using their matching timestamps and request flow, then mitigated the incident by ending the chaos condition and verifying that order submissions could complete successfully again. The main obstacle was that the user-facing error appeared in `jwt-pizza-service`, while the direct root failure occurred in the downstream factory dependency.

## Root cause

The root cause was intentional chaos injection in the pizza factory order fulfillment path, which caused the downstream factory API to intermittently return HTTP 500 responses. `jwt-pizza-service` depends on that factory call to complete an order, so downstream failures propagated to the public `POST /api/order` endpoint.

## Resolution

The service was restored by disabling the chaos condition in the affected dependency or environment, which stopped the factory from returning synthetic `500` errors. Recovery was confirmed by observing that new order requests no longer returned `500` responses and that the order failure alert cleared.

Time-to-mitigation could be improved by documenting the exact chaos toggle location and expected recovery steps in the on-call runbook so the responder can disable the fault immediately without needing to inspect multiple services first.

## Prevention

This incident was a controlled exercise rather than an unexpected production regression, but it exposed a dependency risk that could recur in real incidents: a downstream factory outage causes order creation failures in the public service.

Future incidents with the same pattern can be reduced by strengthening dependency-specific alerting, documenting recovery procedures for chaos toggles, and ensuring dashboards clearly separate application failures from downstream fulfillment failures.
